What is Refactoring?

Code refactoring is the process of changing a program’s source code without modifying its external functional behavior, to improve some of the nonfunctional attributes of the software.

Basically it means, re-writing it so that it will be easier to read, understand and easily adapt to changes in the future. Behavior of the system SHOULD NOT get changed.

Sometimes, Code Refactoring also helps us to cut down the size of the code and improve our performance.

What to keep in mind during refactoring?

The goal of refactoring is NOT to add new functionality The goal is refactoring is to make code easier to maintain in the future If we have testing on the project we should be following Test Driven Development (TDD) and Continuous Integration (CD) during code refactoring. It will greatly minimize the risk of introducing new bugs in the system.

It should be done as a series of small iterations rather than re-writing the whole system again. Each iteration should be responsible for slightly improving the code while leaving the system in working order.

We should not combine Code refactoring with Development of new functionality.

Ideally there should NOT be any new development during Code Refactoring. Minimum, person who is doing code optimization should not work on new feature development for that period of time.

Refactoring According to Standards

Naming Booleans

For Booleans, we want to stick to natural language.

For example, we want to know if a car has a license plate. The result is a Boolean, so we name the variable that holds this Boolean hasLicense. To give another example, we want to know if a person is sick so we use isSick. Both is and has are commonly used prepositions to indicate the variable holds a Boolean.

Looping Arrays

Last, keep things simple when naming variables for looping arrays. Let’s take a look at the below example, which takes a car out of the cars array: 

						
							for (const car of cars) {
								// business logic
							}
Use the above conventions for refactoring your code. These standards help you to make your code more readable.

Meaningful Variable Names

A name should be descriptive and tell a fellow developer what it holds

Meaningful Variable Names

							
							const t = 10; // elapsed time in days
For example, what if we want to store the number of days that have elapsed? We could a simple one-letter variable with a comment. Of course, that’s a bad practice.

Meaningful Variable Names

						
							const elapsedTime = 10;
To further improve the naming of this variable, let’s try using elapsedTime. Again, this is not a very good naming as it does not reveal the type of the variable.

Meaningful Variable Names

						
							const elapsedTimeInDays = 10;
Finally, let’s try one more time using elapsedTimeInDays. This name is much better as it indicates the variable holds a number and explains what the number represents.

Don’t Repeat Yourself

Advantages of DRY Code

  • Maintainability
  • Re-usability
  • Resourceful
  • Ease of Access

Maintaining the code is easier since we already have the logic in place. In case of any update, it will be fast since we have to update only at one place, worst case only at few places only depending on the application complexity.

That piece of code can be re-used any number of times.

Saves time and Space. Initially writing DRY code will take time, however on longer run, it is highly beneficial.

Easier to read, and saves time during the crucial debugging process.

Modularization

  • Shorter functions
  • More explicative names to describe the code’s purpose
  • Less useless comments

We should be organizing our code in multiple modules or functions, where each one is responsible for doing single task. It should be in accordance with Single Responsibility Principle or SRP in short.

If we don’t follow SRP, logic will be scattered around all over the place and it will be very hard to understand.

When Should You Refactor Your Code?

When you see this...

Code Smells

  • Duplicated Code
  • Long Method
  • Large Class
  • Long Parameter List

Duplicated Code bad because if you modify one instance of duplicated code but not the others, you (may) have introduced a bug!

Long Method long methods are more difﬁcult to understand performance concerns with respect to lots of short methods are largely obsolete

Large Class classes try to do too much, which reduces cohesion

Long Parameter List hard to understand, can become inconsistent

Code Smells

  • Divergent Change
  • Lazy Class
  • Speculative Generality
  • Temporary Field

Divergent Change Related to cohesion: symptom: one type of change requires changing one subset of methods; another type of change requires changing another subset

Lazy Class A class that no longer “pays its way” e.g. may be a class that was downsized by a previous refactoring, or represented planned functionality that did not pan out

Speculative Generality “Oh I think we need the ability to do this kind of thing someday”

Temporary Field An attribute of an object is only set in certain circumstances; but an object should need all of its attributes

Code Smells

  • Data Class
  • Refused Bequest
  • Comments (!)

Data Class These are classes that have ﬁelds, getting and setting methods for the ﬁelds, and nothing else; they are data holders, but objects should be about data AND behavior

Refused Bequest A subclass ignores most of the functionality provided by its superclass Subclass may not pass the “IS-A” test

Comments (!) Comments are sometimes used to hide bad code “…comments often are used as a deodorant” (!)
